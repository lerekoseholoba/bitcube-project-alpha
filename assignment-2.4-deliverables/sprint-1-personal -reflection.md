# Sprint 1 — Personal Reflection

## Key Learning

The most important lesson I learned is that sprint work rarely progresses in a perfectly linear way. Even when stories seem well planned, dependencies between backend, frontend, and testing activities can quickly create delays that affect the entire sprint workflow.

## Assumptions That Failed

The assumption that estimation and integration would remain stable throughout the sprint broke down the most. Backend API structures changed during development, which created frontend delays and additional testing effort that was not fully anticipated during sprint planning.

## Behavioural Change

In Sprint 2, I would communicate dependency risks earlier and push for earlier integration testing rather than waiting until most implementation work is complete. This would help identify compatibility issues sooner and reduce last-minute pressure.

## Professional Growth

I need to strengthen my ability to coordinate effectively across different team roles while maintaining consistent communication during implementation changes. Improving technical planning and proactive collaboration would help me contribute more effectively in a real Scrum environment.
